# Лабораторная работа: Резервное копирование и восстановление PostgreSQL

Контейнеры настроены в `docker-compose.yml`.

## Этап 1. Резервное копирование

Подход, реализуемый в наборе скриптов проекта:
- Полные standalone копии делаются командой `pg_basebackup` с опцией включения WAL в состав копии (`-X stream`) и с упаковкой в tar (`-F t`) и сжатием (`-z`).
- В репозитории добавлен скрипт `scripts/backup_run.sh`, который сохраняет архивы в `/backups` на резервном узле и удаляет архивы старше 30 дней (политика для резервного узла).
- Для хранения копий на основном узле добавлен `scripts/primary_retention.sh` (очистка старше 7 дней). Его можно запускать по cron на хосте или в контейнере primary, если настроить cron.

Пример содержимого `scripts/backup_run.sh` (в проекте файл уже есть):
```bash
#!/bin/bash
# записывает в /backups/base_<timestamp>.tar.gz
export PGPASSWORD=secretpassword
pg_basebackup -h pg_primary -U postgres -D - -Ft -z -Xs -P | cat > /backups/base_$(date +%Y%m%d%H%M).tar.gz
# удалить файлы старше 30 дней
find /backups -type f -name 'base_*.tar.gz' -mtime +30 -delete
```

Cron (рекомендация, выполняется на хосте или в контейнере-инициаторе): запуск дважды в сутки

```
# два раза в сутки, в 02:00 и 14:00
0 2,14 * * * PGPASSWORD=secretpassword docker exec pg_backup /scripts/backup_run.sh >> /var/log/pg_backup.log 2>&1
# периодическая очистка на primary (7 дней)
30 2 * * * docker exec pg_primary /scripts/primary_retention.sh >> /var/log/primary_retention.log 2>&1
```

Замечания по реализации и проверке:
- `pg_basebackup -X stream` включает требуемые WAL сегменты в состав полной копии (WAL не архивируются отдельно).
- Исполняемый файл `scripts/backup_run.sh` в репозитории создаёт timestamped tar.gz и удаляет старые архивы (30 дней) — соответствует требованию хранения на резервном узле.
- Для хранения копий на основном узле (`/backups` в контейнере primary) добавьте монтирование каталога в `docker-compose.yml` (в проекте можно смонтировать `./primary_backups:/backups`) и запустите `primary_retention.sh` по cron для удаления старых >7 дней.

### Пересчёт объёма резервных копий за месяц (корректный разбор)

Дано:
- Средний объём новых данных в БД за сутки: 100 МБ (увеличивают размер БД).
- Средний объём изменённых данных за сутки: 1000 МБ — эти изменения генерируют WAL.
- Требуется 2 полных бэкапа в сутки.

Предположения для расчёта (явно):
1. Начальный объём базы S0 — возьмём S0 = 0 для простоты (можно подставить реальный размер).
2. Каждый день размер базы увеличивается на 100 МБ (т.е. после d дней размер Sd = 100*d МБ).
3. Каждый полный бэкап содержит «полный образ» базы на момент бэкапа + WAL, накопленные за период захвата (мы используем stream, поэтому WAL включаются в архив). Предположим, что WAL, соответствующие одному ежедневному набегу изменений 1000 МБ, распределяются равномерно между двумя бэкапами — то есть каждый бэкап в день несёт ~500 МБ WAL.

Расчёт:
- Число бэкапов в месяц: 30 дней * 2 = 60 бэкапов.
- Размер базы меняется линейно от 0 до 3000 МБ за 30 дней. Средний размер = (0 + 3000)/2 = 1500 МБ = 1.5 ГБ.
- Средний размер полного бэкапа (без WAL) ≈ 1.5 ГБ.
- Средний WAL на бэкап ≈ 0.5 ГБ.

Итого средний размер одного бэкапа ≈ 1.5 + 0.5 = 2.0 ГБ.
Общий объём за месяц ≈ 60 * 2.0 ГБ = 120 ГБ.

Вывод: численно получается около 120 ГБ — согласуется с предыдущей грубой оценкой (~123 ГБ). Рекомендация: для более точной оценки нужно знать фактический начальный размер базы и профиль генерации WAL (размеры, частота чекпоинтов, retention конфигурация).

## Этап 2. Потеря основного узла
Сценарий:
1. Останавливаем основной узел: `docker stop pg_primary`
2. Распаковываем на резервном узле базу из бэкапа:
```bash
# на узле pg_backup
rm -rf /var/lib/postgresql/data/*
tar -xzf /backups/base_XXX.tar.gz -C /var/lib/postgresql/data
# Запускаем СУБД 
docker restart pg_backup
```

## Этап 3. Повреждение файлов БД
Удаляем данные таблицы на основном сервере:
1. Получаем OID: `SELECT relfilenode FROM pg_class WHERE relname='test_data';`
2. `rm /var/lib/postgresql/data/base/16384/16395` (здесь 16384 - OID базы, 16395 - OID таблицы).
3. СУБД выдаст ошибку отсутствия файла при выполнении запроса `SELECT * FROM test_data;`.

Восстановление:
Останавливаем сервер primary. Очищаем все данные в директории `data/`.
Распаковываем базовый архив `base.tar.gz` и архив `pg_wal.tar.gz`.
Поскольку табличное пространство (`ts1`) должно быть перенесено (исходная директория недоступна), мы:
1. Создаём новую директорию (`/var/lib/postgresql/tablespace2`).
2. Распаковываем архив `16395.tar.gz` (файл tablespace) туда.
3. Корректируем конфигурацию, создавая symlink: `ln -sfn /var/lib/postgresql/tablespace2 /var/lib/postgresql/data/pg_tblspc/16395`.
4. Меняем владельца всех данных на `postgres`.
5. Запускаем СУБД - данные обоих табличных пространств снова доступны.

## Этап 4. Логическое повреждение данных
1. Добавляем данные: `INSERT INTO my_table VALUES (1, 'good');`
2. Логическое повреждение: `UPDATE my_table SET name = 'garbage';`
3. Делаем дамп схемы или данных с резервного узла (где они сохранились, если это бэкап) или восстанавливаем из бэкапа на резервный узел, поднимаем его, и делаем pg_dump:
```bash
pg_dump -h pg_backup -U postgres -t my_table -a lab_db > restore_table.sql
```
4. Применяем на основном узле:
```bash
# очищаем мусор на основном
psql -U postgres -d lab_db -c "TRUNCATE my_table;"
# восстанавливаем
psql -U postgres -d lab_db -f restore_table.sql
```

docker-compose up -d
